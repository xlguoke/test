# 前端开发者从零搭建完整业务系统的落地方案

我来给你一个**真正能落地、能长期维护**的方案，基于你"熟悉 Node.js、不会后端"的背景，尽量降低心智负担。

---

## 一、技术选型核心思路

作为前端，你应该**避免从零写后端**，而是用以下两条路线之一：

**路线 A：BaaS 优先（推荐你用这条）**
用 Supabase 或 Pocketbase 作为后端底座，你只写业务逻辑和 UI。适合 1 个人 + 长期维护。

**路线 B：全栈框架**
用 Next.js（App Router）+ Prisma + PostgreSQL，自己写 API Route。代码全在一个仓库，前端思维能无缝衔接。

下面我给出 **路线 A（Supabase 方案）** 的完整落地路径，因为它最省事；最后会说什么情况下该切到路线 B。

---

## 二、推荐技术栈（路线 A）

| 层                   | 选型                                                         | 理由                                                         |
| -------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 数据库 + 鉴权 + 存储 | **Supabase**（PostgreSQL + Auth + Storage + Realtime）       | 一站式，控制台点点点就能建表，自带 RLS 权限                  |
| 后台管理 PC 端       | **React + Vite + Ant Design Pro Components** 或 **Refine.dev** | Refine 专为后台定制，CRUD 自动生成                           |
| 移动端               | **uni-app（Vue3）** 或 **Taro（React）**，一次开发多端（H5/小程序/App） | 国内商品+会员场景通常要微信小程序                            |
| 支付 / 充值          | 微信支付 + 支付宝，通过 **Supabase Edge Function（Deno）** 中转 | 支付回调必须有服务端，Edge Function 就是你写的 Node 风格函数 |
| 部署                 | Supabase 云 + Vercel（管理后台）+ 小程序平台                 | 全部托管，不用碰服务器                                       |
| 监控                 | Sentry + Supabase Logs                                       | 前端错误 + 后端日志                                          |

## 三、数据模型（最小可用版）

先把核心表列清楚，这是整个系统的骨架：

- `users` —— Supabase Auth 自带，扩展一张 `profiles` 存会员信息（昵称、手机、头像）
- `members` —— 会员等级、成长值、积分、余额（或合并进 profiles）
- `member_levels` —— 等级定义（青铜/白银/黄金，升级阈值、折扣率）
- `products` —— 商品（名称、图、价格、库存、分类、上下架状态）
- `categories` —— 商品分类
- `orders` + `order_items` —— 订单主表 + 明细
- `recharge_records` —— 充值记录
- `balance_logs` / `growth_logs` —— 余额、成长值流水（**非常重要，所有资产变动必须有流水，后期对账全靠它**）
- `roles` + `permissions` + `role_permissions` + `user_roles` —— RBAC 权限（后台管理员用）
- `operation_logs` —— 后台操作日志

**关键设计原则**：涉及钱和成长值的字段，**永远不要直接 UPDATE**，而是写流水 + 触发器重算余额，或者用数据库事务。这是后期不出事故的底线。

## 四、权限系统怎么做

两层权限要分清：

**C 端会员权限**：用 Supabase 的 RLS（Row Level Security）。比如"用户只能看自己的订单"，一条 SQL 策略搞定，不用写任何后端代码。

**B 端后台权限**：经典 RBAC。角色表 + 权限表 + 角色-权限关联。前端通过 `hasPermission('product:edit')` 控制按钮显隐，后端通过 RLS + Edge Function 二次校验（前端校验绝不能当真）。

## 五、分阶段落地路线图

这是你应该**严格按顺序**执行的，别想着一次性全做完：

**第 0 阶段：打地基（3-5 天）**
- 注册 Supabase、Vercel、微信开放平台账号
- 建 Git 仓库（推荐 monorepo：`apps/admin` + `apps/mobile` + `packages/shared`，用 pnpm workspace）
- Supabase 上把核心表建好，配好 RLS
- 定义好 TypeScript 类型（用 `supabase gen types` 自动生成，前后端类型一致，这是前端做全栈最爽的一点）

**第 1 阶段：后台管理 MVP（2-3 周）**
- 登录 / 权限菜单
- 商品 CRUD、分类管理
- 会员列表、会员等级配置
- 订单列表（只读）
- 用 Refine.dev 能把开发时间砍一半

**第 2 阶段：移动端 MVP（3-4 周）**
- 微信登录 / 手机号登录
- 商品列表、详情、购物车、下单
- 个人中心、订单列表、会员信息展示
- 先不做支付，用"模拟下单"打通全流程

**第 3 阶段：支付和充值（1-2 周）**
- 接微信支付（Edge Function 处理下单签名和回调）
- 充值流程 + 余额流水
- 会员成长值计算规则
- **这一阶段必须做的事**：所有金额计算用 `decimal` 不用 `float`；回调接口必须做幂等（同一个订单号重复回调只处理一次）

**第 4 阶段：稳定性和运营工具（持续）**
- Sentry 接入
- 后台加数据看板（日活、GMV、新增会员）
- 运营位管理（Banner、优惠券基础版）
- 备份策略（Supabase 自带 PITR，但重要数据建议额外导出）

## 六、长期维护的几个关键点

这部分比选型更重要，是很多人栽跟头的地方：

**环境隔离**：至少分 `dev / prod` 两套 Supabase 项目，千万别在生产库上调试。预算够的话加一套 `staging`。

**数据库变更管理**：用 Supabase CLI 的 migration 功能，每次改表结构都生成 SQL 文件提交到 Git。不要在控制台上直接改生产库的结构——这是未来最大的坑。

**CI/CD**：GitHub Actions 连 Vercel 自动部署，小程序用官方 CI 工具自动上传体验版。

**版本灰度**：移动端发版要有灰度能力，小程序天然支持，H5 可以用 Vercel 的分支部署 + 流量切分。

**文档**：哪怕只有你一个人，也要维护一份 `README` + `数据字典` + `接口约定`。半年后的你不会记得今天写的东西。

## 七、什么情况下切到路线 B（Next.js 自建）

- 业务逻辑非常复杂，RLS 写起来比写代码还累
- 需要对接大量第三方系统（ERP、WMS、企业微信等）
- 团队扩充到 3 人以上，开始有专职后端
- 对供应商锁定敏感（不想绑死 Supabase）

切换成本其实不高，因为数据还是 PostgreSQL，搬迁主要是把 RLS 逻辑改写成 API Route。

---

## 八、避坑清单（血泪经验）

一些常见陷阱，简单列一下：不要用前端直接调用微信支付（签名必须在服务端）；不要在 JWT 里放敏感信息；会员余额一定要有流水表，不能只存当前值；商品价格字段用整数存分，不要用浮点；订单状态用状态机管理，不要散落在各处的 if-else；图片必须走 CDN，Supabase Storage 自带；小程序和 H5 的登录态要分开设计，别混着用。
