# 机车俱乐部饮品与社区系统 落地执行方案 v1.0

---

## 一、业务概述与核心目标

**业务定位**：为机车俱乐部提供“饮品售卖 + 会员积分 + 骑行社区”一体化的轻量数字平台。

**实施策略**：  
- 初期聚焦饮品点单、支付与会员积分快速上线；  
- 后续自然扩展社区内容，提升用户粘性与俱乐部文化。

**关键约束与决策**：

| 决策项     | 选定方案                                  | 决策理由                             |
| ---------- | ----------------------------------------- | ------------------------------------ |
| 移动端入口 | **微信公众号 + H5**                       | 社区内容免审核，个体户可上线         |
| 支付通道   | **收钱吧（聚合支付）**                    | 费率低至 0.38%，支持个体户，对接简单 |
| 后端服务   | **uniCloud 阿里云版**                     | 零运维、免费额度、Node.js 生态       |
| 前端框架   | **uni-app（点单 H5）+ Vue3 独立社区项目** | 点单复用成熟组件；社区完全解耦       |
| 成本策略   | 年固定成本控制在 **500 元以内**           | 利用免费额度和低费率支付渠道         |

## 六、收钱吧支付对接方案

### 6.1 前期准备

1. 通过“收钱吧商户版”APP 以个体户资质注册。
2. 开通「微信公众号支付」，绑定已认证服务号 AppID。
3. 配置支付授权目录：`https://m.bikeclub.cn/order/`。
4. 获取商户凭证：`merchant_no`、`app_id`、`app_key`。

### 6.2 统一下单云函数关键流程

1. 接收前台传来的 `goods`、`table_no`、`openid`。
2. 查询会员等级，应用折扣计算实付金额（单位：分）。
3. 生成内部订单号，写入 `orders` 表。
4. 调用收钱吧 JSAPI 统一下单接口，传入 `pay_type=WX_JSAPI`、`sub_openid`。
5. 返回 `jsapi_params` 给前端。

### 6.3 前端调起支付

```
WeixinJSBridge.invoke('getBrandWCPayRequest', jsapiParams, (res) => {
  if (res.err_msg === 'get_brand_wcpay_request:ok') {
    // 支付成功跳转
  }
})
```

### 6.4 支付回调处理

云函数验证收钱吧签名后：

- 更新订单：`pay_status=1, order_status='making'`；
- 计算积分并写入 `point_logs`；
- 更新会员积分与等级。

## 七、多场景消费录入方案

| 场景               | 操作人    | 流程                                       | 订单记录                       |
| ------------------ | --------- | ------------------------------------------ | ------------------------------ |
| 堂食自助           | 顾客      | 扫桌台码 → H5 点单 → 支付                  | source=customer，带桌号        |
| 店员代客           | 店员      | 收银端选品 → 输入/扫描会员 → 现金/余额收款 | source=cashier，记录店员 ID    |
| 微信支付（店员端） | 店员/顾客 | 收银端生成支付二维码，顾客扫码付           | pay_way=wechat，source=cashier |

**会员码**：会员中心生成加密 uid 图片，收银端扫描解析。

## 八、解耦设计与迁移准备

为提高未来可迁移性，从项目初始即封装三个核心服务模块：

**1. 数据访问层 dbService.js**
统一封装所有 MongoDB 操作，云函数中禁止直接调用 `uniCloud.database()`。

**2. 支付服务 paymentService.js**
封装收钱吧下单、验签方法，仅暴露 `createOrder` 和 `verifyCallback` 等接口。

**3. 用户认证 authService.js**
仅暴露 `wxLogin(code)`、`checkToken(token)`，内部可用 uni-id 实现，但业务侧不直接依赖 uni-id 的权限表。

后期迁移时，只需替换这三个文件的实现，业务代码无需改动。

## 九、开发路线与AI协作计划

| 周次 | 核心任务                                                     | 产出物                               |
| ---- | ------------------------------------------------------------ | ------------------------------------ |
| W1   | 公众号认证、收钱吧注册、uniCloud 初始化；搭建 uni-app H5 骨架，跑通微信网页授权 | H5 空白首页可用，用户登录态就绪      |
| W2   | 商品模块云函数与页面；收钱吧下单联调、支付回调与订单生成     | 可完成一笔真实支付                   |
| W3   | 会员积分、等级、积分流水；订单列表与详情；后台管理初版       | 点单端功能闭环，后台可管理商品与订单 |
| W4   | 店员收银端 H5 及云函数；独立社区项目初始化与发帖功能         | 社区雏形，收银功能可用               |
| W5   | UI 打磨、异常处理、桌台码生成、全流程测试、部署上线          | 正式营业版本                         |

**AI辅助原则**：每次编写云函数或页面时，将表结构、业务规则、接口定义作为提示词，让 AI 生成基础代码，开发者负责修改与集成。

## 十一、风险与缓解措施

| 风险点                | 应对策略                                       |
| --------------------- | ---------------------------------------------- |
| 微信网页授权域名限制  | 公众号设置中将域名加入 JS 安全域               |
| 收钱吧回调偶发延迟    | 前端增加主动查单按钮，调用收钱吧查单接口补救   |
| uniCloud 未来收费调整 | 解耦代码已就绪，一周内可迁移至自建 Node 服务   |
| 社区违规内容          | 后台先做人工审核，后期可接入微信内容安全 API   |
| 单人开发精力不足      | 社区功能首期仅上线发图文，评论等功能按反馈迭代 |
